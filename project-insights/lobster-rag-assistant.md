# 龍蝦（RAG 個人記憶助手）專案洞察

> 最後更新：2026-04-10
> 對外發表成熟度：Internal Only
> 專案別名：龍蝦、RAG 個人記憶助手、OpenCore 助理層

## 一頁摘要
- **目前階段**：助理化方向重定義中
- **核心價值主張**：不只回答問題，而是負責接收新文件、整理知識、修正資料庫內容，作為 RAG 之上的助理層。
- **本次更新重點**：
  - 角色從「RAG 本體」轉向「知識維護與資料整理助理」
  - 更明確區分既有 RAG flow 與助理層職責
  - session continuity 與產品定位仍需收斂
- **PM 判斷**：概念是對的，但目前比較像技術原型；要成為產品，必須把使用者工作流與助理價值說得更清楚。

## 專案目標
- 建立一個會持續累積文件記憶、可透過聊天介面查詢的個人助理。 `來源：2026-03-20 逐字稿；2026-03-20 會議記錄`
- 讓新文件進來後，可以自動整理、更新知識庫，並在資料有誤時協助修正。 `來源：2026-04-10 逐字稿`

## 欲解決的痛點
- 大量文件無法每次都直接塞進模型 context 中。 `來源：2026-03-20 逐字稿`
- 使用者希望 AI 可以記住先前文件、對話與 routine，而不是每次重新教。 `來源：2026-03-20 逐字稿`
- 單純把所有事情都丟給 RAG，產品邏輯容易混亂，角色需要重新劃分。 `來源：2026-04-10 逐字稿`

## 目前進展
### 3/20 階段：RAG 基礎能力
- 查詢回答與監看資料夾更新 DB 已可運作。 `來源：2026-03-20 會議記錄`
- 讀檔 → 結構化 → 存 DB 的一次性流程仍需微調。 `來源：2026-03-20 逐字稿；2026-03-20 會議記錄`
- 模型方面暫時偏向雲端模型，以解決本地模型 context 不足。 `來源：2026-03-20 逐字稿；2026-03-20 會議記錄`

### 4/10 階段：角色重新定義
- 團隊重新定義 OpenCore / 龍蝦 與 RAG 的角色分工：RAG 負責知識查詢主流程，助理層負責新文件整理、知識修正與資料更新。 `來源：2026-04-10 逐字稿`
- 若發現資料找不到或 vector 有誤，助理層應協助修正與補資料，而不是直接取代原始 RAG 流程。 `來源：2026-04-10 逐字稿`

## 產品化敘事成熟度
- **目前狀態**：中低
- **已具備**：
  - 文件入庫 / 查詢概念
  - 自動監控資料夾更新
  - 助理層定位的雛形
- **仍欠缺**：
  - session continuity
  - 明確 persona（PM、AE、RD、主管）
  - 產品化敘事：為什麼它不只是 file search
  - 與會議流程、產品頁生成、專案知識庫之間的分工

## MP 狀態
- **是否已 MP**：否
- **目前判斷依據**：目前仍是內部原型與概念驗證，不論角色定義、記憶穩定性或對外敘事都還未成熟。 `來源：2026-03-20 逐字稿；2026-04-10 逐字稿`
- **距離 MP 的缺口**：
  - 助理層 vs RAG 本體的清晰切分
  - 可重現、穩定的記憶與修正流程
  - 面向真實使用者的價值敘事
- **預估時程**：短期適合內部驗證與共用知識工具；距離正式產品化仍有一段距離。 `來源：2026-04-10 逐字稿推定`

## 客戶與目標市場
- 目前較偏內部知識工作、文件整理、個人助理與團隊知識輔助使用情境。 `來源：2026-03-20 逐字稿；2026-04-10 逐字稿`
- 若未來成熟，可延伸到 PM、AE、RD 等知識密集角色。 `來源：2026-04-10 逐字稿推定`

## 產業價值與對外敘事
### 產業價值
- 若內部知識、文件、Wiki、Forum、Log 都能被整理成個人可問答知識庫，這會是很強的 productivity layer。 `來源：2026-03-20 逐字稿推定`

### 差異化亮點
- 不只查詢知識，還能維護知識、補資料與協助修正內容。 `來源：2026-04-10 逐字稿`

### 對外可說版本
- 現階段不建議單獨大幅對外宣傳；更適合先作為內部知識助理與其他工作流（如會議整理、DeviceOn 解法查詢）的底層能力。

## 技術亮點與重大突破
### 技術亮點
- 已具備文件入庫、查詢回答與自動監看資料夾更新 DB 等基礎能力。 `來源：2026-03-20 逐字稿；2026-03-20 會議記錄`
- 助理層定位開始明確：不只是答題，而是幫忙整理新文件、修正知識與更新資料庫。 `來源：2026-04-10 逐字稿`

### 重大突破
- 4/10 的最大進展不是模型本身，而是角色重新定義：RAG 本體負責查詢，助理層負責知識維護。 `來源：2026-04-10 逐字稿`

### 對外可用技術敘事
- 「龍蝦不是另一個單純的 RAG chatbot，而是負責維護知識、吸收新文件並修正內容的助理層。」
- 「它讓知識庫不只會被查詢，也能被持續更新與整理。」

## 技術實踐與落地方式
### 實作方式
- 透過讀檔、結構化整理、資料入庫與 query recall 機制，建立可查詢知識庫。 `來源：2026-03-20 逐字稿`
- 4/10 後的主要方向是把新文件處理與知識修正交給助理層，而不是讓所有查詢都混在同一個 RAG 流程。 `來源：2026-04-10 逐字稿`

### 驗證方式
- 已驗證 query recall 與目錄監看更新 DB 能力。 `來源：2026-03-20 會議記錄`
- 目前主要在驗證角色切分是否能讓產品邏輯更清楚。 `來源：2026-04-10 逐字稿推定`

### 已知限制
- session continuity 與記憶穩定性仍未成熟。 `來源：2026-03-20 會議記錄`
- 產品定位與 persona 仍需更清楚。 `來源：2026-04-10 逐字稿`

## 技術現況與架構重點
- 查詢回答、資料入庫與目錄監看能力已具備基礎。 `來源：2026-03-20 逐字稿；2026-03-20 會議記錄`
- 4/10 後的主要調整，不是新增模型能力，而是重新定義產品角色。 `來源：2026-04-10 逐字稿`

## 風險、阻塞與待確認
- **產品風險**：若定位說不清楚，很容易被誤解為只是另一個 file search 工具。 `來源：2026-04-10 逐字稿推定`
